# 身后协议设计 — 架构总结

## 目标定义

设计一套协议，使 LLM 在用户去世后能以用户人格与关系圈成员持续互动。

**关键决策：选项 B（不透明）** — 对方不知道在与 LLM 交互，体验上尽可能接近"还在"。

---

## 核心问题识别

语感不是主要挑战。**关系知识**才是。

具体说：posthumous LLM 需要知道的不是"你们去过哪里"，而是：
- 这个人在你生命里扮演什么角色
- 你们互动的固定 pattern 是什么
- 当前关系里有哪些未结的线索
- 他/她对你了解什么，不了解什么
- 你对他/她了解什么，不了解什么

流水账（events-minor）已经是高度抽象的产物，仍然无法覆盖吃饭、打游戏这类琐碎但构成关系质地的日常。这是结构性缺口，无法靠人工记录解决。

---

## 转折点：wechat-to-LLM 的真实潜力

之前 wechat-to-LLM 主要用于 ACG 群聊总结。但 chat log 本身就是关系的地面真相——它携带互动 pattern、语气、当前话题、关系纹理。

这把问题从"如何系统性记录关系"转化为"如何设计好的 context 加载机制"。

**架构转变**：posthumous LLM 不依赖静态关系文件来"记住"关系，而是在需要时加载对方的近期 chat log，从中提炼当前关系状态。

---

## 目标架构（三层）

### 数据层 — wechat-to-LLM（现有）
chat log 导出，保持原有能力不变。需保持定期解密更新，确保数据时效性。

### 合成层 — 新增 skill（待开发）
核心工作。读 chat log → 提炼关系模型。

提炼维度：
1. **角色定义**：这个人在用户生活里的位置和功能
2. **互动 pattern**：节奏、触发方式、语气、典型对话结构
3. **当前线索**：关系中正在进行的话题或未结状态
4. **双向知识边界**：对方知道用户什么 / 不知道什么；用户知道对方什么
5. **gap 处理协议**：posthumous LLM 遇到不知道的事时，针对此段关系应采取的策略（绕开 / 用关系惯性填空 / 其他）

### 存储层 — Memex `memory-relation/`（待创建）
合成输出落入 Memex 新目录，与 `memory-semantic/` 同级，成为身份系统的组成部分。
